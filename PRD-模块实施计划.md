## 一、项目概述

### 1.1 项目背景
当前存在多个支持 OpenAI 协议的 AI 站点，不同站点的可用模型、稳定性、响应时间和可维护性差异较大。人工逐个维护站点、逐个测试模型、手动切换调用源的方式成本高、效率低，并且难以形成统一的可视化监控和中转能力。

本项目目标是建设一个基于 `ASP.NET 8 Web` 的统一管理平台，用于完成以下工作：

- 管理多个 OpenAI 协议兼容站点
- 管理平台内置模型库
- 从站点拉取模型并快速导入
- 检测模型在不同站点上的可用性
- 通过可视化方式查看状态、趋势和统计结果
- 对外提供统一的 OpenAI 与 Anthropic 协议中转接口
- 通过优先级切换提升模型整体可用率
- 管理平台访问 Key，并记录调用使用情况
- 记录检测日志与使用日志，支撑排障和运营分析

### 1.2 项目定位
本系统是一个集 **站点管理、模型管理、检测调度、可视化监控、协议中转、Key 管理、日志分析** 于一体的 AI 基础设施管理平台。

### 1.3 产品目标

#### 业务目标
- 降低多站点接入与维护成本
- 提升模型调用稳定性和成功率
- 建立统一的站点与模型可用性观测体系
- 提供标准 OpenAI 与 Anthropic 协议中转能力，减少调用方适配成本

#### 技术目标
- 基于 .NET 8 构建稳定的 Web 管理平台与中转服务
- 使用 SQLite 快速落地单机版部署能力
- 使用 EF Core 降低数据开发复杂度
- 使用 NLog 实现系统日志与业务日志记录
- 管理端首期采用 MVC 或 Razor Pages 方案实现单机版无登录后台
- 支持 OpenAI 与 Anthropic 两类协议的统一兼容与扩展
- 支持后续平滑扩展为更复杂的部署方式

### 1.4 目标用户
- 平台管理员
- 运维人员
- API 调用方
- 内部测试或业务系统对接人员

### 1.5 非目标范围
当前版本暂不强制纳入以下能力，可作为后续增强项：
- 多租户架构
- 完整计费系统
- 分布式部署与高可用集群
- 多协议深度兼容
- 复杂 RBAC 权限体系
- 复杂审批流与审计流

## 二、技术约束与实现原则

### 2.1 技术约束
- 后端框架使用 `ASP.NET 8 Web`
- 数据库使用 `SQLite`
- ORM 使用 `EF Core`
- 系统日志使用 `NLog`
- 对内优先兼容 `OpenAI API` 站点协议，对外兼容 `OpenAI API` 与 `Anthropic API`
- 管理端界面风格要求简洁、美观、清晰

### 2.2 设计原则
- 保持最小可行闭环，优先实现核心业务链路
- 优先复用统一协议封装与检测逻辑，避免多处重复实现
- 管理端与中转服务共用核心领域能力，但逻辑分层清晰
- 检测、路由、中转、日志、统计应尽量解耦
- 重要业务日志必须可追踪、可检索、可清理
- 所有功能必须考虑可扩展性与后续维护成本

## 六、用户场景

### 6.1 站点接入场景
管理员批量导入多个 OpenAI 协议站点地址和 Key，统一启用或禁用，并查看站点基础状态。

### 6.2 模型归集场景
管理员从多个站点拉取模型列表，将新模型一键导入到平台模型库，并建立站点与模型的映射关系。

### 6.3 模型巡检场景
管理员选择某个模型，例如 `gpt-5.4`，对全部启用站点执行一次检测，查看每个站点是否可用、耗时如何、失败原因是什么。

### 6.4 定时监控场景
管理员为指定模型配置定时检测任务，平台自动周期性执行检测，并在图表界面展示 24 小时、7 天、30 天的成功率与响应时间趋势。

### 6.5 对外中转场景
外部调用方向平台发起标准 OpenAI 协议请求，平台根据模型路由规则按优先级选择站点发起请求，若当前站点失败则自动切换到后续候选站点，直到成功返回或最终失败。

### 6.6 日志分析场景
管理员在日志界面查看检测记录和中转使用记录，分析某模型在哪些站点更稳定、某个访问 Key 的请求量和失败率如何。

### 7.8 OpenAI / Anthropic 协议中转模块

#### 7.8.1 目标
对外提供统一的 OpenAI 与 Anthropic 协议兼容接口，并通过站点优先级与失败切换提高可用率。

#### 7.8.2 功能范围
首期建议支持以下接口：
- OpenAI：`GET /v1/models`
- OpenAI：`POST /v1/chat/completions`
- OpenAI：`POST /v1/embeddings`
- Anthropic：`POST /v1/messages`
- Anthropic：`GET /v1/models`

#### 7.8.3 中转流程
1. 外部调用方携带平台访问 Key 发起请求
2. 系统校验 Key 状态
3. 根据请求模型匹配路由规则
4. 获取目标候选站点列表
5. 按优先级依次尝试调用
6. 某站点成功则立即返回结果
7. 某站点失败则继续切换下一个候选站点
8. 全部失败则返回统一错误结果
9. 全流程记录使用日志

#### 7.8.4 路由能力
- 支持外部模型名映射
- 支持多个候选站点
- 支持优先级配置
- 支持候选启用/禁用
- 支持失败后自动切换
- 可扩展熔断、冷却和恢复策略

#### 7.8.5 业务规则
- 被禁用站点不得参与中转
- 路由规则无候选站点时应明确报错
- 请求总超时需可配置
- 需支持普通响应与流式响应两种模式
- 中转接口返回结构应尽量保持 OpenAI 或 Anthropic 协议兼容

#### 7.8.6 验收标准
- 调用方可使用平台接口替代直接访问站点
- 候选站点失败后能自动切换
- 成功结果与失败结果均可记录日志

## 八、非功能需求

### 8.1 性能要求
- 支持多站点并发检测
- 中转请求增加的额外耗时应尽量小
- 图表统计查询在常用时间范围下应保持可接受响应速度
- 批量导入、批量拉取、批量检测应具备合理并发与超时控制

- 中转接口后续应预留限流能力

### 8.4 可维护性要求
- 分层结构清晰
- 站点适配层与业务服务层解耦
- 检测逻辑、中转逻辑、调度逻辑相互独立
- 日志与统计支持后续扩展

### 8.5 UI 要求
- 风格简洁
- 信息层级清晰
- 操作路径短
- 表格、筛选区、图表布局明确
- 支持状态颜色区分成功、失败、禁用、异常等类型

## 十三、核心业务流程

### 13.1 站点接入流程
1. 管理员新增或批量导入站点
2. 系统校验输入格式与去重规则
3. 保存站点信息
4. 管理员可选择启用或禁用站点
5. 后续可发起模型拉取或检测

### 13.2 模型拉取流程
1. 选择一个或多个站点
2. 系统调用各站点模型列表接口
3. 获取返回模型集合
4. 展示结果并标记新旧模型
5. 管理员执行导入
6. 系统写入模型库或建立映射关系

### 13.3 模型检测流程
1. 选择模型和站点
2. 系统按规则发起检测请求
3. 解析结果并判定成功或失败
4. 保存检测日志
5. 更新最近状态与统计结果

### 13.4 定时检测流程
1. 管理员配置任务
2. 调度器按计划触发执行
3. 系统并发检测目标站点
4. 保存执行结果和日志
5. 更新图表统计数据

### 13.5 中转调用流程
1. 外部调用方携带平台访问 Key 发起请求
2. 系统校验 Key 状态
3. 根据请求模型查询路由规则
4. 获取候选站点列表
5. 按优先级依次尝试调用
6. 成功则立即返回
7. 失败则记录链路并继续切换
8. 全部失败则统一返回错误
9. 保存使用日志与调用统计

## 十四、日志与保留策略

### 14.1 系统日志

- 日志中的敏感字段应考虑脱敏或截断

## 十六、异常场景与处理要求

### 16.1 站点异常
- 站点超时
- 站点鉴权失败
- 站点返回格式异常
- 站点不可达

处理要求：
- 明确记录错误信息
- 不中断其他站点检测或切换流程
- 支持后续熔断策略扩展

### 16.2 中转异常
- 无路由规则
- 无可用候选站点
- 所有候选站点都失败
- 平台访问 Key 无效或已过期

处理要求：
- 返回统一错误格式
- 记录完整失败链路
- 日志中标记最终失败原因

### 16.3 日志异常
- 日志写入失败
- 内容过大
- 清理任务异常

处理要求：
- 系统日志记录异常
- 业务流程尽量不因非关键日志失败而整体中断
- 对超大内容进行截断或摘要保存

## 二十、已确认决策与风险说明

### 20.1 已确认决策
- 首期先做单机版无登录后台
- 管理端技术路线采用 MVC 或 Razor Pages，不采用 Blazor Server
- 定时任务方案接受 `Hangfire` 或 `Quartz.NET`
- 对外协议同时支持 OpenAI 与 Anthropic
- 使用日志中的输入与输出按 Token 统计，不保存完整正文
- 中转接口首期支持流式响应
- 平台访问 Key 不限制可访问模型范围
- 站点 API Key 首期不做加密存储
- 检测成功判定不额外增加更严格规则
- UI 风格采用简洁风

### 20.2 当前仍需最终拍板项
- MVC 与 Razor Pages 二选一
- `Hangfire` 与 `Quartz.NET` 二选一
- Anthropic 协议首期除 `/v1/messages` 外，是否还要兼容更多接口

### 20.3 风险点
- 不同站点虽兼容 OpenAI 协议，但返回行为可能存在差异
- OpenAI 与 Anthropic 双协议并行支持会增加适配和测试复杂度
- SQLite 在高并发日志写入场景下能力有限，需关注写入压力
- 流式响应场景下的异常切换、统计汇总和日志落地实现复杂度较高
