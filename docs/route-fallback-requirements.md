# 路由规则需求说明

## 1. 背景

当前路由需求不是“本地模型名映射到单个上游模型名”的简单别名关系，而是要支持一个本地入口模型对应多个上游候选实例，并在调用失败时按顺序自动切换。

典型场景是：

- 已接入一个 CPA 聚合站点，该站点下有很多模型
- 也可能同时接入多个其他站点
- 希望对外只暴露一个统一入口模型，例如 `auto`
- 调用 `auto` 时，不固定绑定某一个上游，而是按配置顺序依次尝试多个上游模型实例

因此，这里的核心目标是：

**给一个本地入口模型配置一组有顺序的候选上游实例，并在调用失败时自动 fallback 到下一个候选。**

---

## 2. 核心目标

需要实现一个**严格顺序执行的 fallback 路由机制**。

示例：

- 本地对外模型：`auto`
- `auto` 下配置多个候选上游实例：
  - 候选1：站点A / 模型X
  - 候选2：站点B / 模型Y
  - 候选3：站点A / 模型Z
  - 候选4：站点C / 模型Q

当用户请求 `auto` 时：

1. 先尝试候选1
2. 候选1失败，则尝试候选2
3. 候选2失败，则尝试候选3
4. 某个候选成功后立即返回结果
5. 只有全部候选都失败时，才返回最终失败

---

## 3. 本地入口模型的含义

这里的 `auto` 只是一个**本地逻辑入口名**，它的作用是统一对外暴露一个模型标识。

它不代表以下含义：

- 不要求对应某个固定上游模型
- 不要求和某个上游模型同名
- 不要求与上游模型建立一一映射关系

正确语义应当是：

- `auto` 代表一组“候选上游模型实例”的统一入口
- 调用方只请求 `auto`
- 系统内部再按照配置顺序去尝试真实上游

也就是说：

**本地入口名只是路由标识，不是上游模型身份。**

---

## 4. 候选上游实例的配置规则

### 4.1 支持任意数量

一个本地入口模型下，可以挂载任意数量的上游模型实例。

不是只能配置一个，也不是固定只能配置少量项，而是支持 N 个候选实例。

### 4.2 支持任意站点

每个候选实例都可以来自任意站点。

例如同一个 `auto` 路由下，可以同时配置：

- 站点A / 模型1
- 站点B / 模型2
- 站点C / 模型3

### 4.3 同一个站点允许重复出现

这是关键要求。

系统不能把“站点”当成唯一项来限制配置。

同一个站点可以在同一个路由中出现多次，例如：

- 站点A / 模型1
- 站点A / 模型2
- 站点A / 模型3

也就是说，路由配置的对象不是“站点列表”，而是：

**有顺序的上游模型实例列表。**

每一项都是一个独立候选实例，站点是否重复不应成为限制条件。

### 4.4 不要求名称对应

本地入口名和上游模型名之间不要求同名，也不要求存在一一对应关系。

例如：

- 本地模型：`auto`
- 上游候选可以是：
  - `siteA/gpt-4`
  - `siteB/claude-sonnet`
  - `siteC/deepseek-chat`

这类配置都应视为合法。

因此，系统不能假设“本地模型名必须等于上游模型名”。

---

## 5. 请求执行逻辑

### 5.1 严格按人工配置顺序执行

候选实例必须严格按照人工配置的顺序串行尝试。

例如当前顺序为：

1. 候选A
2. 候选B
3. 候选C

则请求流程必须是：

- 先请求 A
- A 失败才请求 B
- B 失败才请求 C

系统不能擅自改变该顺序。

### 5.2 成功即停止

一旦某个候选实例返回成功结果，就立即停止后续尝试，并将该结果返回给调用方。

例如：

- A 失败
- B 成功

则直接返回 B 的结果，不再请求 C、D。

### 5.3 全部失败才整体失败

只有当所有候选实例都失败时，整个本地入口模型请求才算最终失败。

---

## 6. 失败判定规则

当前候选实例是否失败，不能只看网络异常，还要看返回结果是否真正可用。

以下情况都应视为当前候选失败，并触发 fallback 到下一个候选：

### 6.1 HTTP 状态码不是 200

只要上游返回的 HTTP 状态码不是 `200`，当前候选就算失败。

### 6.2 空响应

即使接口有响应，但如果返回内容为空、缺少有效结果，仍然应视为失败。

### 6.3 上游业务错误

即使 HTTP 层请求成功，如果上游返回的是业务错误，也应视为失败。

也就是说，真正的判断标准是：

**当前候选是否返回了可用结果。**

只要结果不可用，就应继续尝试下一个候选。

---

## 7. 全部失败时的返回规则

当所有候选实例都失败时，最终应返回：

**最后一个上游的错误**

而不是：

- 统一包装成一个新的通用 fallback 错误
- 返回第一个上游错误
- 返回多个错误的汇总结果

也就是说，最终错误出口要保留“最后一次真实尝试”的错误信息。

---

## 8. 顺序调整要求

候选实例顺序必须支持在管理界面中动态调整。

调整后需要满足：

- 实时生效
- 不需要重建路由
- 不需要重启服务
- 不需要等待额外同步流程

同时，顺序只允许由人工配置控制。

系统不应基于历史成功率、失败率或其他运行结果自动调整候选顺序。

也就是说：

**顺序是人工维护的静态优先级，而不是系统自适应优先级。**

---

## 9. 正确的数据抽象

该需求对应的正确数据关系应当是：

**本地路由入口模型**
→ **一个有序的候选上游实例列表**

例如：

`auto` → `[实例1, 实例2, 实例3, 实例4, ...]`

其中每个候选实例至少需要表达：

- 对应站点
- 对应上游模型
- 在当前队列中的顺序

重点是：

- 列表必须有顺序
- 列表元素是“实例”，不是“站点”
- 允许同一站点重复出现
- 不要求与本地入口模型同名

---

## 10. 明确排除的错误实现

为了避免实现偏差，这里明确说明该需求**不是**以下逻辑：

### 10.1 不是“一个站点只能出现一次”

站点不能作为唯一键限制候选配置。

### 10.2 不是“本地模型名必须匹配上游模型名”

本地入口名只是一个逻辑名称，不是上游模型名称映射。

### 10.3 不是“静态一对一映射”

不是 `auto -> 某一个固定上游模型` 的关系。

### 10.4 不是“负载均衡”

不是多个上游之间做流量分摊。

### 10.5 不是“并发竞速”

不是同时请求多个候选，然后使用最先成功的结果。

### 10.6 不是“自动优化顺序”

不会因为某个候选最近成功率更高，就被系统自动提升优先级。

---

## 11. 一句话总结

该需求本质上是：

**为本地对外模型提供一个可动态维护的、有序上游实例队列，并在请求时严格按人工配置顺序执行 fallback：成功即返回，失败则继续，全部失败时返回最后一个上游错误。**
